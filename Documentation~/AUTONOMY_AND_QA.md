# 自律ループの仕組み と 並行開発でのQA活用ガイド

このドキュメントは、AutoPilotフレームワークの「自律実行ループがどう動くか」と、
「開発中のゲームに対して自動プレイをQAとして使う際の設計上の注意点」を、
別セッション/別プロジェクトからも参照できるようまとめたもの。

アーキテクチャそのものの説明は [FRAMEWORK.md](FRAMEWORK.md) を参照。
本書はその補遺で、(A) 自律ループの内部機構の詳細、(B) 並行開発でのQA設計論、を扱う。

---

## A. 自律ループはどう動くか

### A-1. 「自動でPlay」と「自動でAI起動」は別々の仕組みで、1つのフラグで連結する

2つの自動化が互いを直接呼ばず、`EditorPrefs` の真偽値1個(`AutoPilot.EnableOnPlay`)を
介して疎結合している。この分離のおかげで、手動で `Tools > AutoPilot > Enable On Play` に
チェックして普通にPlayを押しても、まったく同じ経路でAIが起動する(ブリッジ専用の裏道ではない)。

```
【エディタ側】AutoPilotBridgeService        【ランタイム側】AutoPilotBoot
 (エディタプロセスで常駐)                    (Playが始まった"ゲーム"の中で動く)
        │                                          │
   ① Playを自動で押す ──┐          ┌── ② Play開始時に自動で起動
        │                │          │           │
        └─> EditorPrefsに └─ フラグ ─┘ フラグを読んで自分を
            "有効"と書く    EnableOnPlay  起動するか判断
```

### A-2. ① 自動でPlayが始まる(エディタ側)

支えるUnity機能は2つ:

- **`[InitializeOnLoad]` + `EditorApplication.update`**: `[InitializeOnLoad]`の静的コンストラクタは
  エディタがコードをロードするたびに自動実行される。そこで `EditorApplication.update += Poll` を
  登録すると、Pollがエディタの毎ティック呼ばれる常駐ループになる(実際は0.5秒間引き)。
  これがブリッジの心臓で、`command.json` を監視し状態機械を回す。
- **`EditorApplication.EnterPlaymode()`**: コードから「Playボタンを押す」API。

`run`コマンド受信からPlay突入まで:
1. `CheckForCommand()` が新しい `command.json` を検出 → state=waitCompile → `AssetDatabase.Refresh()`
   (= ディスク上のコード変更を取り込んで再コンパイルさせる)
2. Pollが `EditorApplication.isCompiling` を監視して完了待ち。エラーなら status=compile-error で中止
3. `StartPlay()`: 起動シーンを開き、**EditorPrefsのフラグを立て**、`EnterPlaymode()` を呼ぶ

### A-3. ② Play中にAIが自動で走り出す(ランタイム側)

支えるのは **`[RuntimeInitializeOnLoadMethod(AfterSceneLoad)]`**。
Unityが「Play開始後、最初のシーンをロードし終えた直後」に、シーンに何も置いていなくても
自動で呼ぶメソッドを指定する属性。この一点でどのシーンからPlayしてもボットが起動する。

`AutoPilotBoot.Bootstrap()` の流れ:
1. `IsEnabled()` で **エディタ側が立てた EditorPrefs フラグを読む**(握手の受け側)。falseなら即return
2. `Application.runInBackground = true`、セーブ削除(毎回NEW GAME)
3. `VirtualGamepad` 生成 → Input Systemに仮想デバイス追加
4. `BotRunner.Create()`(DontDestroyOnLoadの常駐MonoBehaviour)→ 以後毎フレーム `Update()` が
   `Sensors.Tick → Director.Tick` を回す = AIが走り出す
5. Sensor群 + シナリオを登録

### A-4. 最大の難所: ドメインリロードをまたぐ状態の受け渡し

`EnterPlaymode()` を呼ぶとUnityはドメインリロード(C#全静的状態のリセット)を起こす。
素朴に書くと「Playを押した瞬間にブリッジが記憶喪失になる」。2つの永続ストアで乗り切る:

- **`EditorPrefs`(①→②の握手フラグ)**: レジストリ/plistに保存され、エディタ↔Play境界も
  エディタ再起動も越える。だからリロードの向こう側でランタイムの `IsEnabled()` が確実に読める
- **`SessionState`(状態機械の現在地)**: エディタ起動中は保持される一時ストア。状態・コマンドID・
  終了予定時刻をここに置く。リロード後 `[InitializeOnLoad]` が再実行されてPollを貼り直し、
  SessionStateから「今はplay状態」と読んで続きから再開する

### A-5. Claude Code(エージェント)側はどう自律するか

エージェントが持つ道具は3つだけ: Edit/Write(.csをディスクで書換)、PowerShell(command.json書込・
status.jsonポーリング)、Read(reportを読む)。特別なUnity連携APIは無い。

- **修正が反映される理由**: Editはディスクの実ファイルを即書換。次のrunの1歩目 `AssetDatabase.Refresh()`
  がそれを取り込んで再コンパイルする。エージェントはコンパイルを指示する必要がない
- **再実行と結果確認が自律する理由(心臓部)**: 監視スクリプトを `run_in_background` で起動 → 完了すると
  **ハーネスがエージェントを再呼び出し**(`<task-notification>` を注入)する。これがエージェントの
  "クロック"。常時走っているのではなく、通知で叩き起こされるイベント駆動。監視枠(約9.5分)を超える
  長いrunは、通知のたびに監視を貼り直す(chunk 1/2/3…)
- **どう直すかはLLMの推論**: reportのobjective列で進捗、warn/errorと座標で詰まり箇所を読み、
  失敗→原因→修正を毎回考えてEditする(定型スクリプトではない)

### A-6. エディタ側ブリッジとエージェントは構造的に双子

| | エディタ側ブリッジ | Claude Code |
|---|---|---|
| クロック | `EditorApplication.update` | バックグラウンドタスクの完了通知 |
| 監視対象 | command.json | status.json / report.json |
| 中断をまたぐ記憶 | SessionState | 会話コンテキスト + ディスク上の編集 |
| やること | Play/Refresh を叩く | Edit / command書込 |

2つの自律ループが `AutoPilotBridge/` フォルダを共有メモリにして噛み合っている。

### A-7. 完全無人ではない部分(正直な境界)

- 人間が最初にUnityを開き、1回だけフォーカスしてブリッジ自体をコンパイルさせる必要がある
- エージェントは常駐ではなく通知駆動なので、通知が来ない限り眠っている

## B. 開発と並行して自動プレイをQAに使う際の注意点

このセッションのボットは「完成(開発中止)ゲームをクリアする」目的に最適化されていた。
並行開発でQAとして使うのは**目的が逆**であり、同じ機構が逆の意味を持つ。以下はその設計論。

### B-0. 中心命題: 目的が逆になると、良いボットが悪いボットになる

完成ゲームを「クリアする」ボットと、開発中のゲームを「テストする」ボットは、
**回復(recovery)の扱いが正反対**。クリア用の武器(レスキューテレポート、アイテム付与シム、
迂回リトライ)は、完成ゲームでは賢い回復だが、並行開発では**バグ信号を握りつぶす装置**になる。

象徴例: このセッションのレスキューテレポートは `(-63,-359)` の**同一座標で繰り返し発動**していた。
完成ゲームなら「詰まったので脱出、続行」で正解。だが開発中なら、あの固定座標での反復こそ
「プレイヤーが物理的にハマる地形の穴がここにある」という**レベルデザイナーへの悲鳴**であり、
黙ってワープで越えた瞬間、**本物のコリジョン/NavMeshバグをチームから隠蔽**したことになる。

> 原則: **回復は"成功"ではなく"バグ報告"として扱う。** すべての回復・リトライ・シム・迂回は
> infoログに埋もれさせず、種別と発生場所つきで計上し、集計して人に見せる。

### B-1. そもそも「バグかどうか」をどう知るか(オラクル問題)

「ボットが詰まった」はまだバグではない。バグとは**期待(invariant)の違反**。
期待を3層で持つ:

- **エンジン不変条件**: 未処理例外ゼロ、TransformにNaNが出ない、床を貫通して落ち続けない、
  フレームレートが床を割らない、メモリが単調増加しない
- **ゲーム設計不変条件**: 「このクエストは完了可能」「この目標NPCは到達可能」「HPは0未満にならない」
- **ボット内部条件**: 「私の経路探索は成功」「私の観測は非null」

上2層の違反=バグ、3層目の違反=ボット側不具合の候補。この混同を避けるのが切り分けの土台。

### B-2. 「ゲームのバグ」か「ボットのバグ」かの切り分け

確度の高い順:
- **例外の発生元アセンブリ**: 未処理例外のスタックトレースが**ゲーム側アセンブリ**で始まれば
  ほぼゲームバグ。実例=このセッションで見つけた `DialogueManager` の `ArgumentOutOfRangeException`
  (ゲーム自身のコールバック内で、"速い入力"という実プレイヤーも出せる入力で踏み抜いた)。
  **ボットとゲームをasmdefで分離しておくこと自体が切り分けツール**になる
- **決定性の再現テスト**: 固定シード+同一入力で再現するか。ゲームバグは決定的経路で再現し、
  ボットのflakinessは再現しないことが多い
- **どの層のassertionが鳴ったか**を失敗イベントに必ず記録する

### B-3. 「ボットが本物のバグを直す/回避してしまう」罠(最重要)

- **回復は必ず"場所・状態"に紐づけて集計**し、同一地点での反復回復が「ホットスポット」として
  自動で浮上する仕組みにする(`(-63,-359)`の反復テレポート=レベル欠陥1件として起票されるべき)
- **2モードを持つ**: *進行モード*(回復して深部到達。回帰スイート用)と *監査モード*(回復せず
  fail-fastし、詰まった正確な状態を報告)。**監査モードを持たないとバグは永遠に見えない**
- **ゲーム状態を書き換えて回避しない**: アイテム付与・テレポート・privateフィールド書込は
  ボットがゲームの内臓に手を突っ込む行為。テスト対象を変え、まさにその欠陥を隠す。シムが
  許されるのは**既知でチケット化済みのコンテンツ欠落のみ**、必ずフラグ付きで、「純粋実行」で
  全シムを無効化して**どこで死ぬかを報告できる**ようゲートする
- **actuation(操作)にリフレクションを使わない**: private直呼びやリフレクション操作は
  (a)ゲームのリファクタで壊れて偽陽性を出し、(b)**UIが決して許さない状態遷移を強制できてしまう**
  ため、実プレイヤーには到達不能な"バグ"を報告しかねない。原則=**操作はプレイヤーと同じ入力面
  (仮想パッド)から、リフレクションは観測専用に**(デバイスレベル注入方式が最初から正しい部分)

### B-4. 2つのコードベースが互いの信号を汚染しないために

- **ボットはゲーム改変を要求しない**(するなら最小限の明示されたテストフックだけ)
- **観測フックと"チート"を峻別**: 読み取り専用の観測フックは安全。製品挙動を変えるフック
  (カットシーンスキップ、アイテム付与)は出荷ビルドから排除し明示的に印を付ける
- **ボットの世界モデルはゲームと同じ真実源(データアセット)から導出する**: 手作業で静的抽出した
  接続グラフ(例: ChopChopWorldMap)は日々変わる開発で即腐り、"ゲームが変わった"のか"バグ"なのか
  区別できなくなる。クエストグラフやロケーション接続はゲームが実際にロードする同じ
  ScriptableObjectから読み、**ゲームと共進化**させる
- **ゲーム変更直後のボット失敗は、まず"ボットの陳腐化"を疑う**

### B-5. プロセス・組織面

- **偽陽性率を低く保つ**: 失敗キューがボットのflakinessだらけだと人は見なくなり(アラート疲れ)、
  本物のバグが埋もれる。ボットの信頼性=バグ検出の信頼性
- **失敗時に人間が再現できる証拠を残す**: シード+入力トレース+スクリーンショット+状態スナップ
  ショットを失敗の瞬間に保存。ボット無しで再現できて初めてバグ報告として成立する
- **早期からCIで毎ビルド回す**: 後付けの大掛かりQAにしない。毎日回せばゲーム変更時にボットが
  大きな音で壊れ、モデル陳腐化にすぐ気づける
- **重大度で分類**: エンジン例外 > ソフトロック(目標到達不能) > 見た目の不具合
- **尻尾が犬を振らないよう警戒**: 開発者が「ボットを黙らせるため」に直したり、ボットの期待に
  合わせてゲームを書き始めたら主従逆転。ボットはプレイヤーの代理であり、ボットの都合は目的ではない

### B-6. まとめの一文

**「クリアする良いボット」を作る技術と「バグを見つける良いボット」を作る技術は、
回復の扱いを"沈黙の成功"から"騒がしい欠陥報告"へ反転させない限り、正反対になる。**

並行開発に持ち込むなら、クリア用の全機構を反転させる: 回復は場所付きで起票、シムは既知欠落にのみ
ゲート付きで、操作は入力面から、観測モデルはゲームのデータから導出、そして「回復を切って正直に
死ぬ」監査モードを一級市民にする。切り分けの土台は、ボットが独立した期待(3層の不変条件)を持ち、
失敗を発生元アセンブリと決定的再現で属性付けできること。
